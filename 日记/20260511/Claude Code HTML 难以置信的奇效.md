# 使用 Claude Code：HTML 难以置信的奇效

> 来源：宝玉 @dotey 对 Thariq @trq212 文章的中文整理/翻译  
> 原文链接：<https://x.com/trq212/status/2052809885763747935>  
> 示例链接：<https://thariqs.github.io/html-effectiveness/>  
> 整理日期：2026-05-11

## 一句话总结

随着 AI Agent 能力增强，Markdown 正在逐渐成为复杂输出的束缚；相比之下，HTML 更适合作为 Claude Code / AI Agent 输出复杂信息、实施计划、代码审查、研究报告、设计原型和交互式工具的载体。

## 核心观点

过去 AI Agent 常用 Markdown 和人沟通，因为它简单、便携、容易编辑，也具备基础富文本能力。

但现在 AI Agent 的输出越来越复杂，常见内容包括：

- 长篇需求文档
- 实施计划
- 代码审查说明
- 研究报告
- 设计方案
- 可交互原型
- 数据分析结果

如果继续用 Markdown 展示这些内容，很容易出现以下问题：

- 内容冗长，不容易读完
- 缺少视觉层次
- 分享不方便
- 难以表达复杂图表和交互
- 人很难持续参与审查和决策

因此作者更推荐让 Claude Code 直接生成 HTML 文件。

## 为什么 HTML 更适合 AI Agent 输出

### 1. 信息密度更高

HTML 能承载比 Markdown 丰富得多的信息形式：

- 表格
- CSS 样式
- SVG 图表
- 代码片段
- 图片
- 工作流图
- 交互组件
- 可视化布局
- JavaScript 动态效果

作者的判断是：只要 Claude 能理解的信息，几乎都可以用 HTML 高效表达出来。

相比之下，Markdown 经常迫使模型用低效方式表达复杂内容，例如：

- 用 ASCII 画图
- 用 Unicode 模拟颜色
- 用超长列表堆信息

### 2. 视觉清晰度更高，更容易读

Claude 生成的规格文档和实施计划越来越复杂，经常超过 100 行 Markdown。这样的内容很难让人认真读完。

HTML 可以通过这些方式提升可读性：

- 选项卡
- 卡片布局
- 分区导航
- 插图
- 链接
- 颜色编码
- 响应式布局

核心价值：把复杂信息组织成更容易浏览、理解和审查的形态。

### 3. 更容易分享

Markdown 文件分享起来不方便：

- 浏览器默认渲染不好
- 经常只能当附件发
- 对非技术成员不友好

HTML 文件只要上传到云端，比如 S3，就可以变成一个链接。团队成员可以直接在浏览器打开，移动端也能看。

作者认为：如果 PR 说明、需求文档、分析报告用 HTML 呈现，别人认真阅读的概率会明显提升。

### 4. 支持双向交互

HTML 不只是展示内容，还可以让用户操作。

例如可以让 Claude 生成：

- 滑块
- 旋钮
- 可拖拽卡片
- 表单
- 实时预览
- 参数调节器
- 一键复制按钮

这意味着 HTML 不只是“文档”，还可以是一个临时工具或小型工作台。

一个很有用的技巧：在 HTML 页面中加入“复制为 JSON / Markdown / Prompt / Diff”的按钮。这样可以在页面里完成交互调整，再把结果粘回 Claude Code 继续处理。

## 适合用 HTML 的场景

### 1. 需求、计划与探索

不要只让 Claude 输出一份 Markdown 计划，而是让它生成一个 HTML 思考空间。

适合用于：

- 比较多个实现方案
- 展示数据流
- 画 UI 草图
- 放代码片段
- 做方案权衡
- 生成实施计划

示例提示词：

```text
请生成一个 HTML 文件，展示 6 种不同的新手引导页方案，在布局、语气和信息密度上做出差异，并标注每种方案的取舍。
```

### 2. 代码审查与代码理解

HTML 很适合展示复杂代码逻辑：

- 渲染真实 diff
- 加行内注释
- 用颜色标注严重程度
- 画模块结构图
- 画数据流图
- 解释背压、缓存、限流等复杂机制

作者提到，他现在提交 PR 时经常会附带一个 HTML 格式的代码解读页面。

示例提示词：

```text
帮我审查这个 PR，生成一个 HTML 页面解释它的逻辑。请重点分析数据流和背压逻辑，渲染真实 diff，并用颜色区分问题严重程度。
```

### 3. 设计与原型

HTML 非常适合表达设计想法。

可以用于：

- 生成 UI 原型
- 展示设计系统组件
- 调整动画参数
- 预览交互效果
- 对比多个视觉方案
- 后续再转成 React、Swift 等实现

示例提示词：

```text
请为新的结账按钮做一个 HTML 交互原型。点击后播放动画并变成紫色。加几个滑块，让我调节动画速度、颜色和缓动曲线，并提供复制参数按钮。
```

### 4. 报告、研究与学习

Claude Code 可以读取大量上下文，例如：

- 本地代码库
- Git 历史
- Slack 聊天记录
- Linear 任务
- 浏览器信息
- MCP 接入的数据源

然后生成可视化 HTML 报告。

适合用于：

- 技术调研报告
- 功能原理讲解
- 故障复盘
- 周报
- 管理层汇报
- 架构图
- SVG 流程图

示例提示词：

```text
请阅读限流器相关代码，并生成一个单页 HTML 讲解文档，包含令牌桶流程图、核心代码注释和常见陷阱。
```

### 5. 临时可视化编辑器

有些任务用文字描述很痛苦，可以让 Claude 生成一个一次性 HTML 工具。

例如：

- 任务优先级排序
- 测试用例分类
- 用户反馈打标签
- Feature Flag 编辑器
- Prompt 调试器
- JSON/YAML 配置编辑器
- 正则表达式调试
- Cron 表达式选择
- 动画曲线调节
- 颜色选择
- 裁剪区域选择

核心做法：页面负责可视化操作，最后导出结构化结果，再粘回 Claude。

示例提示词：

```text
请把这 30 个 Linear 任务做成一个可拖拽的 HTML 看板，分为 Now / Next / Later / Cut 四列。最后提供“复制为 Markdown”按钮，导出分类结果和理由。
```

## 实践建议

不需要一开始就搞复杂的 `/html` 技能或模板。

最简单的做法就是直接对 Claude Code 说：

```text
给我做一个 HTML 文件。
```

或者：

```text
生成一个 HTML artifact。
```

关键不是格式本身，而是提前想清楚：

1. 这个 HTML 是给谁看的？
2. 它要帮助我理解什么？
3. 是否需要交互？
4. 是否需要导出结果？
5. 是否要用于团队分享？

## 常见疑问

### HTML 会不会更耗 Token？

会。Markdown 通常更省 token。

但作者认为，HTML 带来的表现力和阅读率提升，足以抵消 token 成本。尤其在大上下文模型里，多出来的 token 成本可以接受。

### HTML 会不会生成更慢？

会。作者估计可能比 Markdown 慢 2 到 4 倍，但结果通常值得等待。

### HTML 的缺点是什么？

主要是版本控制不友好。HTML diff 通常很乱，不如 Markdown 清爽。

### 如何让 HTML 不丑？

可以让 Claude 扫描代码库，生成一份符合公司风格的“设计系统 HTML 文件”。之后再把这个设计系统作为参考，让 Claude 生成风格一致的页面。

## 对我的启发

这篇文章的重点不是简单地说“HTML 比 Markdown 高级”，而是指出：

> AI Agent 输出正在从“文本回答”变成“可视化工作成果”。

Markdown 适合轻量沟通；HTML 更适合复杂协作。

在 Claude Code 这类工具里，HTML 可以成为：

- 需求说明书
- 代码审查页面
- 数据报告
- 临时工具
- 设计原型
- 交互式决策界面
- 团队共享资料

作者最后的观点是：HTML 让他感觉自己重新回到了人机协同的循环中，而不是面对几百行 Markdown 文档时只能盲目信任 AI。

## 可复用提示词模板

```text
请基于以下资料生成一个单页 HTML artifact，用于帮助我快速理解和分享。
要求：
1. 用清晰的视觉层级组织信息。
2. 包含核心结论、背景、关键概念、流程/结构图和行动建议。
3. 对重要风险或取舍用颜色标注。
4. 如果适合交互，请加入筛选、折叠、滑块或一键复制按钮。
5. 页面要适合团队成员直接打开阅读。
```
