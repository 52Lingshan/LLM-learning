Bailing V3 从模型对比到 Megatron-Bridge 适配

这份页面把 Tiny vs Flash 架构差异和 Megatron-Bridge 适配工作合并成一个讲解型界面。 重点不是罗列参数，而是回答三个问题：Flash 为什么更大更快？Bailing V3 的 MLA/KDA/MoE 如何排布？适配训练框架时为什么会出现 NaN，又该如何定位？

15.5× Flash 相对 Tiny 的权重规模约放大倍数
5:1 Flash 每 6 层中 KDA:MLA 的比例
512 Flash routed experts 数量
42+1 Flash 主干层数 + MTP 层

先给结论

Bailing V3 Flash 不是 Tiny 的简单放大版。它在三个层面变了：模型规模显著变大，KDA 占比更高，KDA gate 计算从外置变成 kernel 内置融合。Megatron-Bridge 适配的核心工作，就是把这种混合架构准确映射到 MCore 的异构 attention、MoE、MTP 和权重转换体系里。

Model

Flash 更大，也更偏向 KDA

Flash 是 42 层、hidden size 2560、512 experts。层组从 Tiny 的 4 层一组变成 6 层一组，因此 KDA:MLA 从 3:1 提升到 5:1。

Kernel

Flash 把 gate 融进 KDA kernel

Tiny 先调用 fused_kda_gate 计算 gate，再把结果传给 KDA。Flash 传入 A_log 和 dt_bias，由 kernel 内部完成 gate。

Bridge

适配难点在权重映射

NaN 的根因不是数据坏了，而是部分 expert 和 MLA Q LoRA 权重没有被正确加载。修复重点是补齐 SequentialMLP 和 Q LoRA 映射。

Tiny vs Flash：不是只差参数量

Tiny 和 Flash 共享 Bailing V3 的大范式：MLA + KDA + MoE + MTP。但 Flash 的规模、层排布、专家数量和 kernel 调用路径都发生了关键变化。

hidden_size 1536 2560 主干宽度更大，单层容量更强。

关键提醒： Tiny vs Flash 文档中写到 Flash 的 q_lora_rank=null；Megatron-Bridge 适配摘要中又列出 Bailing V3 的 MLA 层有 q_lora_rank=768。 这可能对应不同配置快照或适配过程中使用的 Flash 权重版本。实际转换时必须以当前 checkpoint 的 config.json 为准。

Bailing V3 的混合架构怎么运转

Bailing V3 的主干不是单一 attention，而是交替使用 KDA 和 MLA。Flash 的典型排布是每 6 层一个周期：5 层 KDA，1 层 MLA。前 2 层 MLP 是 Dense，之后进入 MoE。

为什么 KDA 占比高

KDA 是线性注意力路径，更适合吞吐和长序列效率；MLA 是 softmax 注意力路径，用来保留更强的精确注意力能力。 Flash 把比例调成 5:1，本质是更激进地把主干计算推向 KDA，再周期性插入 MLA 做能力补偿。

MoE 在哪里

除前 2 层 Dense MLP 外，后续层使用 MoE。Flash 有 512 个 routed experts，每个 token 选择 8 个专家，同时有 shared expert。

KDA kernel 差异：Flash 的名字从哪里来

Tiny 与 Flash 建模代码只有 42 行差异，核心集中在 BailingMoeV3KimiDeltaAttention。最重要的变化是 gate 计算从 Python 侧外置，变成 CUDA kernel 内置融合。 

# Tiny 路径：先在外部计算 gate
g = self.f_a_proj(hidden_states)
g = fused_kda_gate(
    g,
    self.A_log.view(1, 1, -1, 1),
    self.head_dim,
    g_bias=self.dt_bias,
)

o, recurrent_state = chunk_kda(
    q=q,
    k=k,
    v=v,
    g=g,
    beta=beta,
    initial_state=recurrent_state,
    output_final_state=True,
    use_qk_l2norm_in_kernel=True,
    safe_gate=self.safe_gate,
    lower_bound=self.lower_bound,
    cu_seqlens=cu_seqlens,
)
# Flash 路径：把 gate 计算融合进 kernel
g = self.f_proj(hidden_states)
g = rearrange(g, "... (h d) -> ... h d", d=self.head_dim)

o, recurrent_state = chunk_kda(
    q=q,
    k=k,
    v=v,
    g=g,
    beta=beta,
    initial_state=recurrent_state,
    output_final_state=True,
    use_qk_l2norm_in_kernel=True,
    A_log=self.A_log,
    dt_bias=self.dt_bias,
    use_gate_in_kernel=True,
    cu_seqlens=cu_seqlens,
)
Tiny 投影后先调用 fused_kda_gate。
中间结果 gate 张量写回显存。
再调用 KDA chunk_kda 读取已经算好的 gate。
Flash 传入 A_log 与 dt_bias。
融合 kernel 内部完成 gate，减少 launch 和读写。
一句话： Tiny 是“先算 gate，再做 KDA”；Flash 是“把 gate 计算塞进 KDA kernel 一起做”。这就是 Flash 更强调 fused kernel 优化的核心证据。

Megatron-Bridge 适配：把 HF 模型翻译成 MCore 世界

适配工作可以看作一次“结构翻译”：HF 里的 BailingMoeV3ForCausalLM 要映射到 Megatron-LM / Megatron-Bridge 的 transformer config、模块 spec、权重命名和并行策略中。

1. Config 放宽 MLA + rotary_interleaved 限制，加入 kda variant。
2. KDA Module 新增 KimiDeltaAttention，对齐 HF 的分离投影。
3. Module Spec 让 MCore 能构造 MLA/KDA 混合层。
4. Bridge 注册 BailingMoeV3，写参数映射。
5. Recipe 补齐 pretrain 配方与训练参数。

NaN 调试故事：不是训练炸了，是权重没进来

适配里最有价值的部分，是两次 NaN 的根因定位。它说明大模型框架迁移时，最危险的不是显式报错，而是“权重映射静默跳过”。

Problem 1

Expert 权重未加载导致 NaN

现象：iteration 1 loss 为 NaN。Forward dump 显示 experts.local_experts.0.linear_fc1 输出首次出现 NaN，而输入正常。

根因：recipe 设置 moe_grouped_gemm=False，模型使用 SequentialMLP 命名 local_experts.E.linear_fc1.weight，但 bridge 只覆盖了 TEGroupedMLP 的 linear_fc1.weight* 格式，expert 权重被静默跳过。

Fix 1

补齐 SequentialMLP 专家映射

GatedMLPMapping(
    megatron_param="decoder.layers.*.mlp.experts.local_experts.*.linear_fc1.weight",
    gate="model.layers.*.mlp.experts.*.gate_proj.weight",
    up="model.layers.*.mlp.experts.*.up_proj.weight",
)

AutoMapping(
    megatron_param="decoder.layers.*.mlp.experts.local_experts.*.linear_fc2.weight",
    hf_param="model.layers.*.mlp.experts.*.down_proj.weight",
)
Problem 2

MLA Q LoRA 权重未加载导致 NaN

现象：修复 expert 映射后仍 NaN。权重检查发现 linear_q_down_proj.weight 在所有 rank 上为 NaN。

根因：Bailing V3 的 MLA 层使用 Q LoRA 压缩，Megatron 参数是 linear_q_down_proj 和 linear_q_up_proj，但 bridge 只映射了非 LoRA 路径的 linear_q_proj。

Fix 2

补齐 MLA Q LoRA 映射

"decoder.layers.*.self_attention.linear_q_down_proj.weight":
    "model.layers.*.attention.q_a_proj.weight",

"decoder.layers.*.self_attention.linear_q_up_proj.weight":
    "model.layers.*.attention.q_b_proj.weight",

"decoder.layers.*.self_attention.linear_q_up_proj.layer_norm_weight":
    "model.layers.*.attention.q_a_layernorm.weight",

后续验证清单

适配工作已经覆盖主体路径，但文档里也列出了需要继续验证的边界问题。下面按风险从高到低整理。

事项 为什么重要 建议验证
MLA g_proj 门控 MCore MLA 当前不支持 attention_output_gate，可能抛 NotImplementedError。 决定是实现输出门控，还是暂时跳过 g_proj 权重。
KDA TP 分片 当前 KDA 使用简单 AutoMapping，结构匹配但仍要验证 tensor parallel 切分。 跑 bridge 转换 UT，比较 shard 后的参数形状和数值。
异构 layer spec 混合 MLA + KDA 不能只用普通 GPT decoder spec。 确认 provider 使用 get_transformer_block_with_experimental_attention_variant_spec。
MTP 层映射 Layer 42 使用 MLA + MoE，且有独立 eh/enorm/hnorm/final_layernorm。 检查 MTP 层参数加载完整性，并跑 forward。
真实权重 pretrain mock data 只能验证路径，真实权重才能暴露数值和并行问题。 按远程路径加载 Flash checkpoint，运行短步数训练。

术语速查

术语 解释 在本文中的作用
KDA Kimi Delta Attention，线性注意力变体。 Bailing V3 Flash 的主力 attention，5/6 层使用。
MLA Multi-head Latent Attention，带 KV 低秩压缩的 softmax attention。 每 6 层插入 1 层，保留精确注意力能力。
MoE Mixture of Experts，专家混合结构。 Flash 有 512 routed experts，每 token 激活 8 个。
MTP Multi-token Prediction。 Flash 有 1 个 MTP layer，层索引 42。
SequentialMLP 非 grouped gemm 的专家实现，参数名含 local_experts.N。 第一次 NaN 的根因之一，原 bridge 未映射该命名。
Q LoRA Q 投影低秩降维再升维。 第二次 NaN 的根因之一，LoRA 路径权重未加载。
EP / CP / PP Expert Parallelism / Context Parallelism / Pipeline Parallelism。 Flash 路径名中出现 ep8cp8pp4，说明训练使用组合并行。
